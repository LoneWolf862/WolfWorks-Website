Project

Pong Game

Embedded Systems, Real-Time Programming, Graphics, and Autonomous Control

Completed / Functional Coursework Project

Short Description

A Pong game written in C for the MSP432 LaunchPad featuring both player-controlled and autonomous gameplay modes. The project included custom display control, collision detection, real-time game logic, and an AI opponent capable of predicting ball trajectories and automatically positioning its paddle.

Project Overview

The Pong Game project was one of my earliest introductions to applied embedded programming in C. Developed on the MSP432 LaunchPad platform, the project combined graphics generation, real-time game logic, input handling, and autonomous control into a complete embedded application.

The system was built around a small pixel-based graphical display connected to the MSP432. Rather than relying on a prebuilt graphics engine, we had to directly interface with the display hardware and implement the software necessary to draw game objects and update the screen in real time.

The game supported two operating modes. The first was a traditional player-controlled Pong implementation where users could move the paddles and play the game normally. The second was an autonomous mode where the software calculated the future trajectory of the ball and automatically moved the paddles into position to intercept it.

To make gameplay more interesting, I added a feature that increased the ball's speed after every successful hit. This gradually increased the difficulty of the game and eventually pushed the limits of the display hardware. In autonomous mode, the game could become so fast that the display would visibly struggle to keep up, occasionally appearing to black out as update rates exceeded what the display could comfortably refresh.

Features

  • MSP432 LaunchPad implementation.
  • Written entirely in C.
  • Pixel-based graphical display.
  • Custom display control software.
  • Real-time game loop.
  • Player-controlled gameplay mode.
  • Autonomous gameplay mode.
  • Ball and paddle collision detection.
  • Dynamic speed scaling.
  • Automatic trajectory prediction.
  • Autonomous paddle positioning.
  • Embedded user input handling.

Technical Challenges

  • Embedded graphics: interfacing directly with a small pixel display and creating software capable of drawing and updating game elements.
  • Real-time programming: maintaining a responsive game loop while updating graphics and handling user input.
  • Collision detection: accurately determining ball impacts against moving paddles and screen boundaries.
  • Trajectory prediction: calculating future ball movement so autonomous paddles could react appropriately.
  • Performance limits: managing increasing game speeds while remaining within the refresh capabilities of the display hardware.
  • System integration: combining graphics, timing, input handling, game-state management, and AI behavior into a single embedded application.

Miscellaneous Notes

While Pong itself is a simple game, the project represented one of my first opportunities to build a complete embedded application rather than isolated exercises or demonstrations. It required combining multiple concepts into a cohesive system that users could directly interact with.

The autonomous mode became one of the most entertaining aspects of the project. Because the ball accelerated after every successful hit, the AI players could eventually keep the ball in play indefinitely. As the speed continued increasing, the display would become the limiting factor rather than the game logic itself.

Looking back, this project provided early experience with embedded graphics, state machines, timing, software architecture, and hardware interfacing— concepts that would later appear repeatedly in larger and more advanced embedded systems projects.
