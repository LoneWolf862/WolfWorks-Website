Project

Autonomous LiDAR Terrain Mapping Drone

Embedded Systems, Robotics, Telemetry, and 3D Mapping

Archived / Functional Prototype — ECEN 361 project, later continued independently

Short Description

A LiDAR-equipped hexacopter platform designed to scan terrain from the air and generate usable 3D point-map data. The system combined flight control, long-range telemetry, LiDAR scanning, servo motion, onboard logging, and later autonomous GPS-based mission flight.

Project Overview

This project started as an ambitious ECEN 361 drone mapping project. The core concept was to mount a long-distance scanning LiDAR module under a drone, sweep it back and forth with a servo, and combine distance, angle, and timing data into a point-map representation of the terrain below.

The project used an ArduPilot Cube Orange flight controller, LW20 LightWare LiDAR, long-range RFD 900x-US telemetry, a Taranis/FrSky radio control system, and multiple embedded controllers for data collection and control translation. The drone began as a nonfunctional inherited frame and was rebuilt into a mostly stable flying platform before the LiDAR payload and data system were added.

By the end of the semester the drone was operational at a basic level. After the course ended, additional work improved stability, added GPS, enabled more consistent autonomous mission flights, and produced viable raw LiDAR point-map data.

Features

  • Hexacopter airframe rebuilt from a nonfunctional previous project platform.
  • ArduPilot Cube Orange flight controller with loosely tuned, later improved flight control.
  • LW20 LightWare LiDAR module mounted for terrain distance scanning.
  • Servo-driven scan motion to sweep the LiDAR across the terrain.
  • Long-range RFD 900x-US telemetry radio link for mission-control communication.
  • Arduino PWM-to-digital bridge for triggering scan behavior from the flight controller.
  • Embedded data capture through microcontroller firmware and SD card logging.
  • FAA Remote ID telemetry module broadcasting ID, position, altitude, speed, heading, takeoff location, emergency status, and timestamp.
  • Later GPS integration for more consistent autonomous grid-pattern flight.

Technical Challenges

  • Recovering the drone platform: the inherited drone was mostly nonfunctional, so motors, ESCs, power systems, and flight-control hardware had to be evaluated, repaired, or replaced.
  • Learning ArduPilot/Cube Orange quickly: the team had limited experience with the Cube platform, requiring rapid learning of flight-controller setup, control loops, telemetry, and mission behavior.
  • Flight stability: the first working configuration could hover, but loose control tuning made precision flight difficult until later feedback tuning improved stability.
  • Regulatory compliance: the drone’s size and weight pushed it into a category requiring Remote ID compliance, so an ASTM F3411 Remote ID telemetry module was acquired, configured, and installed.
  • Crash repair: a low-altitude wind-related crash destroyed telemetry-radio antenna connections. The antenna connection was repaired using a soldered jumper and nonconductive epoxy reinforcement.
  • LiDAR initialization: the system needed a way to start scanning while airborne, which led to the Arduino PWM-to-digital control bridge.
  • Data path constraints: full radio passthrough was not ready during the semester, so SD card logging was used first, with later work moving data over the wireless radio.
  • Power distribution: the drone required stable 3.3 V, 5 V, and 12 V rails from a 22 V battery system, built using buck converters and added filtering.

Miscellaneous Notes

The original concept was a long-range autonomous terrain scanner that could be launched from a base camp, fly a planned route, and relay useful terrain data over a long-range radio link. The radio system was selected with the hope of supporting telemetry over very long distances.

The final data flow used the handheld controller and receiver to communicate with the Cube flight controller. The Cube handled flight behavior and output PWM signals. An Arduino translated PWM into digital command flags, while the LiDAR controller drove the servo, recorded scan data, and either stored data locally or relayed it through the telemetry radio.
